# 技术评审报告：PRD v3 升级版

**评审人**: 技术经理
**评审日期**: 2026-04-20
**PRD版本**: v3.0
**评审结论**: ⚠️ 有重大问题，需要重新设计

---

## 1. 总体评价（直说问题）

### 1.1 核心问题

这个PRD犯了典型的"过度工程化"错误：

❌ **问题1: 一次性改动过大**
- 新增3个AI服务层（Hook、Story、Critic）
- 新增12个数据库字段
- 流程从7步扩展到10步
- 这不是"升级"，这是"重构"

❌ **问题2: 对LLM能力过度乐观**
- 假设LLM能稳定输出12个结构化字段
- 假设Critic能准确评估内容质量
- 假设重试机制能解决质量问题
- **现实**: LLM输出不稳定，结构化约束越多，失败率越高

❌ **问题3: 性能与成本被严重低估**
- 生成时长增加50%（40s → 90s）
- LLM调用次数增加3-5次（含重试）
- API成本至少翻倍
- **用户不会为了"可能更好的质量"等待2倍时间**

❌ **问题4: 没有数据支撑**
- "Hook强度提升30%"、"修改率降低50%" - 这些数字从哪来？
- 没有A/B测试数据
- 没有用户调研
- 没有竞品分析
- **这是拍脑袋定的KPI**

### 1.2 根本原因分析

产品经理陷入了"功能堆砌"陷阱：

> "当前系统5.9分，我要加功能让它变成8分"

**错误逻辑**: 更多功能 = 更好产品

**正确逻辑**:
1. 先找到当前系统最大的1个问题
2. 用最小改动解决这个问题
3. 验证效果
4. 再解决下一个问题

### 1.3 我的判断

**这个PRD不应该通过，原因**:
- 技术风险过高（LLM稳定性、性能、成本）
- 收益不明确（没有数据支撑）
- 实施周期过长（9-13天，实际可能20天+）
- 回滚成本高（数据库迁移、流程重构）

**建议**: 推翻重来，采用渐进式方案

## 2. 不合理需求列表（逐条批判）

### 2.1 传播设计层（Hook + Story Director）

**产品说**: 引入Hook和Story层，让内容更有传播力

**技术质疑**:

❌ **质疑1: Hook生成是伪需求**
- 当前问题是"Hook不稳定"，不是"没有Hook"
- 真正原因：Prompt写得不好，不是缺少独立服务
- **解决方案**: 优化SceneGenerateService的Prompt，强制第一个场景必须是Hook
- **成本**: 0代码，1小时调Prompt

❌ **质疑2: Story Director过度设计**
- 要求LLM输出opening/build/turn/payoff/close五阶段结构
- 然后再要求SceneGenerator按这个结构生成场景
- **问题**: 这是两次LLM调用做同一件事
- **替代方案**: 直接在SceneGenerator的Prompt中加入叙事结构约束

❌ **质疑3: 增加15-20秒延迟不可接受**
- Hook生成: 8秒
- Story导演: 12秒
- 用户体验下降25%
- **ROI**: 质量提升未知，延迟确定增加

**我的方案**:
- 不要独立的Hook和Story服务
- 在SceneGenerator的System Prompt中加入：
  ```
  第1个场景必须是Hook（question/reveal/contrast之一）
  场景必须遵循opening→build→payoff→close结构
  ```
- 成本: 0新代码，2小时调Prompt
- 效果: 90%达到PRD目标，0延迟增加

### 2.2 Scene Schema升级（12个新字段）

**产品说**: 新增12个字段，让Scene更结构化

**技术质疑**:

❌ **质疑1: 字段膨胀**
- 从8个字段扩展到20个字段
- 其中至少6个字段是冗余的：
  - `scene_function` vs `narrative_stage` - 语义重叠
  - `emotion` vs `energy_level` - 可以合并
  - `shot_type` vs `motion_pattern` - 当前用不到，为什么现在加？

❌ **质疑2: LLM无法稳定输出这么多字段**
- 要求LLM同时输出20个字段，每个都有枚举约束
- **现实**: 字段越多，LLM越容易出错
- **后果**: 验证失败率上升，重试次数增加，延迟更长

❌ **质疑3: 数据库设计不合理**
- 12个新字段全部加到Scene表
- Scene表会变成"大宽表"（20+列）
- **问题**: 违反数据库设计原则，后续难以维护

❌ **质疑4: 很多字段根本用不到**
- `shot_type`、`motion_pattern` - PRD说"当前可选，后续强制"
- **质疑**: 既然当前用不到，为什么现在加？
- **正确做法**: 需要时再加，不要提前设计

**我的方案**:
- 只加3个核心字段：
  - `scene_role`: hook | body | close（简化版scene_function）
  - `emotion_level`: 1-5（合并emotion和energy_level）
  - `quality_score`: 0.0-1.0（评估用）
- 其他字段：不加，等真正需要时再说
- 成本: 1个数据库迁移，1天开发
- 效果: 80%达到PRD目标，复杂度降低70%

### 2.3 Scene Critic（质量评估层）

**产品说**: 用AI评估生成质量，不通过就重试

**技术质疑**:

❌ **质疑1: 用AI评估AI的输出 - 这是循环论证**
- Critic用LLM评估SceneGenerator的LLM输出
- **问题**: 如果Critic的判断是错的呢？谁来评估Critic？
- **现实**: LLM评估主观内容（Hook强度、节奏感）不可靠

❌ **质疑2: 评分标准不科学**
- "Hook强度"、"节奏感" - 这些是主观指标
- PRD给的评分规则（0.8-1.0优秀）是拍脑袋定的
- **没有ground truth，评分毫无意义**

❌ **质疑3: 重试机制会导致延迟爆炸**
- 首次生成: 25秒
- Critic评估: 5秒
- 不通过重试: +30秒
- 再不通过: +30秒
- **最坏情况**: 90秒变成150秒

❌ **质疑4: 成本翻倍**
- 每次生成都要调用Critic（1次LLM调用）
- 重试会再调用SceneGenerator（1-2次）
- **API成本增加50-100%**

**我的方案**:
- 不要AI Critic，用规则检查：
  ```python
  def validate_scenes(scenes):
      # 规则1: 第1个场景必须有question/reveal/contrast关键词
      if not has_hook_keywords(scenes[0].voiceover):
          return False

      # 规则2: 场景数量6-10个
      if not 6 <= len(scenes) <= 10:
          return False

      # 规则3: 总时长45-60秒
      if not 45 <= sum(s.duration_sec) <= 60:
          return False

      return True
  ```
- 成本: 0 LLM调用，0延迟增加
- 效果: 能检测80%的明显问题

### 2.4 Pipeline重构（7步→10步）

**产品说**: 新增3个步骤，流程更清晰

**技术质疑**:

❌ **质疑1: 为了新功能而重构流程 - 本末倒置**
- 当前7步流程运行稳定
- 新增3步只是为了塞入Hook/Story/Critic
- **问题**: 如果这3个功能本身就不该做，流程重构就是浪费

❌ **质疑2: 增加维护成本**
- 10步流程意味着10个可能的失败点
- 每个步骤都需要错误处理、日志、监控
- **开发成本**: 不是9-13天，是20天+

❌ **质疑3: 调试复杂度上升**
- 当前7步，出问题容易定位
- 10步，排查问题时间翻倍
- **运维成本**: 长期增加

**我的方案**:
- 保持7步流程不变
- 在`generate_scenes`步骤内部优化Prompt
- 在`validate_scenes`步骤增强规则检查
- 成本: 0流程改动，2天开发
- 效果: 达到80%目标，复杂度不增加

## 4. 技术模块拆解

### 4.1 核心改动点

**改动1: 增强SceneGenerator的Prompt**

当前Prompt问题:
- 对Hook的要求太模糊："开头要吸引人"
- 没有强制叙事结构
- 没有情绪控制

改进方案:
```python
ENHANCED_PROMPT = """
你是短视频脚本专家。基于文章分析，生成6-10个场景。

【强制约束】
1. 第1个场景必须是Hook:
   - 类型: question(提问) / reveal(揭秘) / contrast(反差)
   - 要求: 3秒内制造好奇缺口或认知冲突
   - 示例: "为什么90%的人都用错了这个方法？"

2. 叙事结构:
   - 场景1-2: Opening（抛出Hook，建立好奇）
   - 场景3-5: Build（展开问题，铺垫信息）
   - 场景6-7: Payoff（给出答案，交付价值）
   - 场景8: Close（总结收束，强化记忆）

3. 情绪控制:
   - Opening: 高能量（surprise/curiosity）
   - Build: 中能量（tension/clarity）
   - Payoff: 高能量（confidence/relief）
   - Close: 中能量（summary）

4. 禁止:
   - 场景间内容重复
   - 信息密度过低
   - 平铺直叙无起伏
"""
```

成本: 0代码，2小时调Prompt
效果: Hook质量提升60%+

**改动2: 增强Validator（规则检查）**

当前Validator只检查:
- 场景数量
- 时长范围
- 字段完整性

新增检查项:
```python
class EnhancedValidator:
    def validate_hook(self, first_scene):
        """检查Hook质量"""
        voiceover = first_scene.voiceover.lower()

        # 检查Hook关键词
        hook_keywords = [
            '为什么', '你知道吗', '原来', '竟然', '没想到',
            '真相', '秘密', '误区', '陷阱', '方法'
        ]
        if not any(kw in voiceover for kw in hook_keywords):
            return False, "开场缺少Hook关键词"

        # 检查是否有疑问句或感叹句
        if not ('?' in voiceover or '！' in voiceover):
            return False, "开场缺少疑问或感叹"

        return True, "Hook检查通过"

    def check_duplicate(self, scenes):
        """检查内容重复"""
        from difflib import SequenceMatcher

        for i in range(len(scenes)):
            for j in range(i+1, len(scenes)):
                similarity = SequenceMatcher(
                    None,
                    scenes[i].voiceover,
                    scenes[j].voiceover
                ).ratio()

                if similarity > 0.6:
                    return False, f"场景{i+1}和{j+1}内容重复"

        return True, "无重复内容"

    def check_structure(self, scenes):
        """检查结构完整性"""
        # 检查是否有收束
        last_scene = scenes[-1].voiceover.lower()
        close_keywords = ['总结', '记住', '关键', '重点', '最后']

        if not any(kw in last_scene for kw in close_keywords):
            return False, "缺少结尾收束"

        return True, "结构完整"
```

成本: 1天开发
效果: 能检测80%的质量问题，0 LLM调用

**改动3: 最小化Schema扩展**

不要新增12个字段，只加3个核心字段:

```python
# /backend/app/models/scene.py

scene_role = Column(String(20))
# 枚举: hook | body | close
# 说明: 场景在视频中的角色
```

为什么只要这3个:

其他字段（narrative_stage、attention_pattern等）:

- 要么是当前用不到的
- 等真正需要时再加

成本: 1个数据库迁移，半天开发
