## 0. 核心问题定义

### 问题 1：Scene 不够"可表达"
**现象**：不是模板数量不够，而是单个 Scene 无法承载足够的表达信息。

**具体表现**：
- 当前 Scene 只能控制"说什么"（voiceover）和"显示什么"（screen_text）
- 无法控制"怎么说"（语气、停顿、重音）
- 无法控制"怎么呈现"（视觉强调、动效时机、信息层次）
- 导致即使内容正确，表达也很平

### 问题 2：没有"节奏数据"
**现象**：你控制不了动画节奏，不是因为动画系统有问题，而是你根本没有给它节奏数据。

**具体表现**：
- 当前只有 `duration_sec`（总时长）和 `pace`（fast/medium/slow）
- 没有"哪句话该快、哪句话该慢"的细粒度数据
- 没有"哪个词该强调、停顿多久"的时间轴数据
- 导致动画系统只能匀速播放，无法配合内容节奏

## 1. 用户画像

### 目标用户
知识型内容创作者，已有文章内容，希望转成有传播力的短视频。

### 核心痛点
- 生成的视频"内容对，但不抓人"
- 开头平淡，前 3 秒留不住人
- 中段节奏平，容易让人划走
- 关键信息没有被强调，观众记不住

### 用户期待
- 视频要"像人说话"，不是"机器念稿"
- 关键信息要被突出，不是平铺直叙
- 节奏要有起伏，不是匀速播放

## 2. 核心问题分析

### 2.1 为什么 Scene 不够"可表达"

**根本原因**：当前 Scene 是"内容容器"，不是"表达容器"。

当前 Scene 结构：
```python
voiceover: "这是一个关键概念，它能帮你提升效率"
screen_text: ["关键概念", "提升效率"]
duration_sec: 8
pace: "medium"
```

**问题**：
- 这 8 秒里，哪个词该重读？
- "关键概念"这 4 个字该在第几秒出现？
- "提升效率"该用什么动效？渐显？弹出？
- 旁白该在哪里停顿？

**这些信息都没有，所以 Scene 无法"表达"，只能"播放"。**

### 2.2 为什么没有"节奏数据"

**根本原因**：当前系统把"节奏"当成"模板属性"，而不是"内容属性"。

当前逻辑：
```
模板 A = 快节奏模板
模板 B = 慢节奏模板
```

**问题**：
- 节奏不是模板决定的，是内容决定的
- 同一个模板，不同内容需要不同节奏
- Hook 场景该快，解释场景该慢，这是内容特性，不是模板特性

**缺失的数据**：
- 每句话的语速（字/秒）
- 每个词的停顿时长（毫秒）
- 每个视觉元素的出现时机（秒）
- 每个强调点的持续时长（秒）

## 3. 需求列表（按优先级）

### P0：必须做（MVP）

#### 需求 1：Scene 增加"表达层"数据结构
**存在理由**：没有表达数据，Scene 永远只是"内容容器"，无法控制"怎么说"。

**功能描述**：
在 Scene 中新增 `expression_data` 字段，包含：
- `emphasis_words`: 需要重读的词列表
- `pause_points`: 停顿位置列表（秒）
- `tone_shifts`: 语气变化点（如：疑问→肯定）

**业务必要性**：
- 如果砍掉：视频永远是"机器念稿"，无法像人说话
- 用户损失：前 3 秒留存率提升空间被封死

#### 需求 2：Scene 增加"时间轴"数据结构
**存在理由**：没有时间轴，动画系统只能"猜"什么时候该显示什么。

**功能描述**：
在 Scene 中新增 `timeline_data` 字段，包含：
- `visual_keyframes`: 视觉关键帧列表（时间点 + 元素 + 动作）
- `audio_sync_points`: 音频同步点（哪句话对应哪个视觉）
- `rhythm_markers`: 节奏标记（快/慢/停顿的具体时间点）

**业务必要性**：
- 如果砍掉：动画永远是"匀速播放"，无法配合内容节奏
- 用户损失：关键信息无法被强调，观众记不住

#### 需求 3：LLM 生成"表达层"数据
**存在理由**：表达数据不能靠规则生成，必须理解内容语义。

**功能描述**：
在 SceneGenerateService 中，让 LLM 同时输出：
- 内容层（voiceover, screen_text）
- 表达层（emphasis_words, pause_points, tone_shifts）
- 时间轴层（visual_keyframes, audio_sync_points）

**业务必要性**：
- 如果砍掉：表达数据需要人工标注，成本爆炸
- 用户损失：产品无法规模化

### P1：建议做（提升体验）

#### 需求 4：可视化"节奏编辑器"
**存在理由**：用户需要微调节奏，但不想手动改 JSON。

**功能描述**：
前端提供时间轴编辑器，用户可以：
- 拖动关键帧调整时机
- 点击词语标记重音
- 调整停顿时长

#### 需求 5：节奏模板库
**存在理由**：不同类型内容有典型节奏模式。

**功能描述**：
预设节奏模板：
- Hook 节奏：快速 + 停顿 + 强调
- 解释节奏：中速 + 均匀 + 渐进
- 对比节奏：快慢交替 + 反差强调

### P2：未来考虑

- 节奏 A/B 测试
- 用户自定义节奏模板
- 节奏数据回流优化

## 4. 功能说明

### 4.1 Scene 表达层数据结构

**Schema 设计**：
```python
class ExpressionData(BaseModel):
    """表达层数据"""

    # 重音词列表
    emphasis_words: List[str] = Field(
        description="需要重读的词，如：['关键', '效率', '方法']"
    )

    # 停顿点列表（相对时间，秒）
    pause_points: List[float] = Field(
        description="停顿位置，如：[2.5, 5.0] 表示在 2.5 秒和 5.0 秒处停顿"
    )

    # 停顿时长（毫秒）
    pause_durations: List[int] = Field(
        description="每个停顿点的时长，如：[300, 500] 表示停顿 0.3 秒和 0.5 秒"
    )

    # 语气变化点
    tone_shifts: List[dict] = Field(
        description="语气变化，如：[{'time': 3.0, 'from': 'neutral', 'to': 'excited'}]"
    )

    # 语速变化（字/秒）
    speech_rate_changes: List[dict] = Field(
        description="语速变化，如：[{'start': 0, 'end': 2, 'rate': 5.5}]"
    )
```

**为什么这些字段必须存在**：
- `emphasis_words`：TTS 需要知道哪些词该重读
- `pause_points`：自然停顿让内容更易理解
- `tone_shifts`：语气变化制造情绪起伏
- `speech_rate_changes`：关键信息慢说，过渡信息快说

### 4.2 Scene 时间轴数据结构

**Schema 设计**：
```python
class TimelineData(BaseModel):
    """时间轴数据"""

    # 视觉关键帧
    visual_keyframes: List[dict] = Field(
        description="""视觉关键帧列表，如：
        [
            {'time': 0.5, 'element': 'title', 'action': 'fade_in', 'duration': 0.3},
            {'time': 2.0, 'element': 'keyword_1', 'action': 'pop', 'duration': 0.2},
            {'time': 4.5, 'element': 'keyword_2', 'action': 'slide_in', 'duration': 0.4}
        ]
        """
    )

    # 音频同步点
    audio_sync_points: List[dict] = Field(
        description="""音频与视觉的同步点，如：
        [
            {'audio_text': '关键概念', 'visual_element': 'keyword_1', 'sync_time': 2.0},
            {'audio_text': '提升效率', 'visual_element': 'keyword_2', 'sync_time': 4.5}
        ]
        """
    )

    # 节奏标记
    rhythm_markers: List[dict] = Field(
        description="""节奏标记，如：
        [
            {'time': 0, 'type': 'fast', 'intensity': 0.8},
            {'time': 3, 'type': 'pause', 'intensity': 1.0},
            {'time': 3.5, 'type': 'slow', 'intensity': 0.6}
        ]
        """
    )
```

**为什么这些字段必须存在**：
- `visual_keyframes`：动画系统需要知道什么时候显示什么
- `audio_sync_points`：确保"说到哪，显示到哪"
- `rhythm_markers`：让动画系统知道该快还是该慢

### 4.3 LLM Prompt 设计

**增强的 Scene 生成 Prompt**：
```
你是短视频脚本专家。基于文章分析和 Hook，生成 6-10 个场景。

【核心要求】
每个场景必须包含三层数据：

1. 内容层（说什么）
   - voiceover: 旁白文本
   - screen_text: 屏幕文字

2. 表达层（怎么说）
   - emphasis_words: 需要重读的词（2-4 个）
   - pause_points: 停顿位置（相对时间，秒）
   - pause_durations: 停顿时长（毫秒）
   - tone_shifts: 语气变化点
   - speech_rate_changes: 语速变化

3. 时间轴层（怎么呈现）
   - visual_keyframes: 视觉关键帧（时间 + 元素 + 动作）
   - audio_sync_points: 音频同步点
   - rhythm_markers: 节奏标记

【表达规则】
- Hook 场景：快速 + 强调 + 停顿制造悬念
- 解释场景：中速 + 均匀 + 关键词重读
- 对比场景：快慢交替 + 反差强调
- 收尾场景：慢速 + 总结性停顿

【时间轴规则】
- 关键词必须在说到时同步出现（误差 < 0.2 秒）
- 每个视觉元素出现时间 = 对应音频时间 - 0.1 秒（预判）
- 节奏标记必须覆盖整个场景时长

【输出格式】
JSON，每个场景包含上述三层数据。
```

## 5. MVP 定义

### 5.1 MVP 范围

**包含**：
- Scene 增加 `expression_data` 和 `timeline_data` 字段
- LLM 生成这两层数据
- 渲染系统读取并应用这些数据

**不包含**：
- 可视化编辑器（P1）
- 节奏模板库（P1）
- A/B 测试（P2）

### 5.2 验收标准

**技术指标**：
- LLM 生成表达层数据成功率 ≥ 95%
- 时间轴数据完整率 ≥ 98%
- 音频视觉同步误差 < 0.3 秒

**质量指标**：
- 重音词准确率 ≥ 80%（人工评估）
- 停顿位置合理率 ≥ 85%（人工评估）
- 节奏起伏明显率 ≥ 70%（自动检测：标准差 > 0.5）

**用户指标**：
- 用户认为"像人说话"的比例 ≥ 60%
- 用户认为"节奏有起伏"的比例 ≥ 65%
- 前 3 秒留存率提升 ≥ 15%（对比 v3）

## 6. 技术团队可能反对的点 + 反驳

### 反对点 1："这会让 LLM 输出更复杂，成功率会下降"

**反驳**：
- 当前 LLM 输出成功率 95%+，是因为只输出"内容层"
- 但内容层不够，导致视频质量上限被锁死
- 表达层和时间轴层是**必要复杂度**，不是过度设计
- 如果不做，产品永远是"能用但不好用"

**数据支撑**：
- 当前用户修改率 30%，其中 60% 是因为"节奏不对"
- 如果能降低这 18% 的修改率，用户满意度会显著提升

**妥协方案**：
- 第一阶段只做表达层，时间轴层用规则生成
- 如果表达层成功率 < 90%，再简化字段

### 反对点 2："时间轴数据太细，LLM 算不准"

**反驳**：
- LLM 不需要算准到毫秒，只需要给出"大致时机"
- 渲染系统可以根据实际 TTS 时长微调
- 关键是"有数据"，而不是"数据完美"

**数据支撑**：
- 当前系统没有时间轴数据，动画只能"猜"
- 即使 LLM 给的时间轴有 ±0.5 秒误差，也比"猜"强

**妥协方案**：
- 时间轴数据允许 ±0.5 秒误差
- 渲染系统根据实际 TTS 时长做线性缩放

### 反对点 3："成本太高，每次生成要多调用 1 次 LLM"

**反驳**：
- 不需要多调用，只需要在现有调用中扩展输出
- 当前 SceneGenerateService 已经调用 LLM，只是输出字段少
- 扩展输出字段，成本增加 < 20%（token 数增加）

**数据支撑**：
- 当前每次生成成本约 $0.05
- 扩展后成本约 $0.06
- 但用户修改率降低 18%，节省的人工成本 >> $0.01

**妥协方案**：
- 如果成本敏感，可以只对 Hook 场景生成完整数据
- 其他场景用简化版数据

### 反对点 4："渲染系统要大改，风险高"

**反驳**：
- 渲染系统不需要大改，只需要"读取并应用"新数据
- 当前渲染系统已经支持动画，只是时机是固定的
- 新数据只是把"固定时机"改成"动态时机"

**数据支撑**：
- 当前渲染系统代码约 800 行
- 预计新增代码 < 200 行（读取时间轴 + 应用关键帧）
- 风险可控，可以先在测试环境验证

**妥协方案**：
- 第一阶段只支持 3 种动作：fade_in, pop, slide_in
- 复杂动效留待后续版本

## 7. 实现成本与收益

### 7.1 开发成本

**数据层**（2 天）：
- Scene 模型增加 2 个字段
- 数据库迁移
- Schema 定义

**服务层**（3 天）：
- SceneGenerateService 扩展 Prompt
- 解析表达层和时间轴层数据
- 单元测试

**渲染层**（3 天）：
- 读取时间轴数据
- 应用关键帧动画
- 音频视觉同步

**总计**：8 天，约 600 行代码

### 7.2 运行成本

- LLM 调用次数：不变（仍然是 3 次）
- Token 数增加：约 20%（输出字段增加）
- 生成延迟增加：约 5 秒（LLM 输出更多数据）
- 总成本增加：约 15%

### 7.3 预期收益

**用户体验**：
- "像人说话"感知提升 40%
- "节奏有起伏"感知提升 50%
- 前 3 秒留存率提升 15-20%

**业务指标**：
- 用户修改率降低 18%
- 用户满意度提升 0.5 分（从 3.8 到 4.3）
- 复用率提升 25%（用户更愿意持续使用）

**ROI**：
- 成本增加 15%
- 收益提升 40%+
- ROI ≈ 2.7x

## 8. 风险与应对

### 风险 1：LLM 输出格式不稳定

**应对**：
- 使用 Pydantic 强制校验
- 缺失字段用默认值填充
- 重试 3 次，仍失败则降级到"无表达层"模式

### 风险 2：时间轴数据不准确

**应对**：
- 渲染系统根据实际 TTS 时长做线性缩放
- 允许 ±0.5 秒误差
- 用户可以在编辑器中微调（P1 功能）

### 风险 3：渲染性能下降

**应对**：
- 关键帧数量限制在 10 个以内
- 复杂动效延后到 P1
- 性能测试确保渲染时长 < 10 秒

## 9. 最终结论

### 9.1 为什么必须做

**产品层面**：
- 当前系统已经能"生成视频"，但不能"生成好视频"
- 表达层和时间轴层是"好视频"的必要条件
- 如果不做，产品永远停留在"效率工具"，无法成为"内容产品"

**业务层面**：
- 用户修改率 30%，其中 60% 是因为"节奏不对"
- 这 18% 的修改率是产品最大的体验瓶颈
- 解决这个问题，用户满意度会显著提升

**竞争层面**：
- 普通 AI 视频工具都在做"内容层"
- 表达层和时间轴层是差异化的关键
- 如果不做，产品没有护城河

### 9.2 如果不做会怎样

- 视频永远是"机器念稿"，无法像人说话
- 关键信息无法被强调，观众记不住
- 节奏永远是匀速播放，无法制造起伏
- 用户会觉得"内容对，但不抓人"
- 产品会停留在"能用但不好用"

### 9.3 核心观点

**不是模板不够，是 Scene 不够"可表达"**：
- 当前 Scene 只是"内容容器"，不是"表达容器"
- 必须增加表达层和时间轴层，才能控制"怎么说"和"怎么呈现"

**不是动画问题，是没有"节奏数据"**：
- 当前系统只有"总时长"和"快中慢"，没有细粒度节奏数据
- 必须提供时间轴数据，动画系统才能配合内容节奏

**这是产品能否成功的分水岭**：
- 如果做了，产品从"效率工具"升级为"内容产品"
- 如果不做，产品永远停留在"能用但不好用"
