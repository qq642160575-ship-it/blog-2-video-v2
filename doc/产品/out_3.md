# AI 博文生成视频系统 PRD v3（冻结版）

## 1. 项目背景

本项目要解决的问题不是“把文字转成视频文件”，而是把一篇已有文章快速转成适合短视频传播的动态讲解视频初稿。

为什么现在做：
- 技术博主和知识博主已经有稳定文章产出，但文章转视频的人工成本仍然高，通常需要 3 到 6 小时。
- 现有工具大多只能做字幕朗读或素材拼接，缺少“文章理解 -> 分镜 -> 模板化动态表达 -> 成片导出”的闭环。
- 当前技术条件已经足够支撑一个收敛版 MVP：LLM 负责结构化理解和分镜，固定模板负责稳定表达，Remotion 负责视频渲染。

本版本目标只有一个：
- 让用户在 10 分钟内，把一篇结构清晰的文章生成一条 45 到 60 秒、9:16、带配音和字幕的 MP4 初版视频。

---

## 2. 用户场景

### 目标用户
- 技术博主
- 知识型内容创作者

本版本不服务：
- 自媒体团队批量生产
- AI 内容创业者的规模化生产需求

### 关键使用路径 1：技术博主生成视频初稿
1. 用户粘贴一篇 Markdown 或纯文本文章。
2. 系统分析文章并生成 6 到 10 个 Scene。
3. 系统使用固定模板生成配音、字幕和 MP4。
4. 用户预览视频，修改个别 Scene 文案或模板类型。
5. 用户重新渲染整片，导出最终 MP4。

### 关键使用路径 2：知识博主调整视频表达
1. 用户提交解释型文章。
2. 系统生成分镜和视频初稿。
3. 用户修改开头钩子和某几个 Scene 的屏幕文案。
4. 系统重新渲染整片并输出新版视频。

---

## 3. MVP 范围

### ✅ 本版本包含

仅包含以下 P0 功能：
- 文章输入：支持纯文本和 Markdown 粘贴
- 文章解析：生成结构化摘要
- 分镜生成：生成 6 到 10 个 Scene
- 固定模板渲染：仅 3 个模板
- TTS 配音：仅 1 个供应商
- 句级或短语级字幕生成
- 9:16 MP4 渲染导出
- Scene 基础编辑
- 编辑后整片重渲染
- 任务状态查询
- 中间产物存储和错误日志存储
- 后端硬性质量校验

### ❌ 本版本不包含

以下功能已明确砍掉：
- 文章链接输入：砍掉。原因是会引入抓取、正文抽取、反爬、版权和异常清洗问题，和 MVP 核心价值无关。
- 5 到 8 种视觉模板：砍掉。原因是模板数量过多会显著提高开发和测试成本，且降低成片稳定性。
- 复杂 Visual DSL：砍掉。原因是抽象成本高，本版本只需要轻量 `SceneSpec` 即可支撑 3 个模板。
- LLM 直接生成 React / Remotion 代码：砍掉。原因是不可控、不可维护。
- Scene 级局部重生成：砍掉。原因是会引入时间轴、字幕、TTS、缓存一致性复杂度；本版本只支持编辑后整片重渲染。
- 批量处理 UI：砍掉。原因是当前目标用户是个人创作者，不是团队生产。
- 多平台版本：砍掉。原因是会拉高分辨率、布局和时长适配成本。
- 横屏视频：砍掉。原因是 MVP 只验证竖屏短视频场景。
- 多套品牌系统：砍掉。原因是当前不服务企业级一致性需求。
- A/B 开头钩子：砍掉。原因是收益低于实现复杂度。
- 自动封面标题和发布文案：砍掉。原因是非核心链路。
- 数据看板：砍掉。原因是对当前闭环无帮助。
- 素材混剪和外部素材检索：砍掉。原因是会引入版权、素材选择和风格一致性问题。
- 30 秒和 90 秒双时长支持：砍掉。原因是节奏逻辑不同，本版本统一固定在 45 到 60 秒。

| 功能 | 是否保留 | 原因 | 优先级 |

| 复杂 Visual DSL | 否 | 过度抽象，不是当前阶段必须品 | - |

## 4. 功能定义

### 功能 1：文章输入

**功能描述**
- 用户通过表单提交文章标题和正文内容。
- 支持 `text` 和 `markdown` 两种输入类型。

**Input**
```json
{
  "title": "什么是 RAG",
  "source_type": "markdown",
  "content": "# 什么是 RAG\n\nRAG 是..."
}
```

**Output**
```json
{
  "project_id": "proj_xxx",
  "status": "created",
  "article_stats": {
    "char_count": 2140,
    "estimated_reading_sec": 320
  }
}
```

**行为规则**
- 支持输入类型仅限 `text` 和 `markdown`。
- 正文长度建议范围为 1000 到 3000 中文字。
- 少于 500 字时，不进入生成，返回“内容不足，建议补充论点”。
- 超过 3000 字时，提示超限；本版本不做长文自动分块。
- 输入内容必须保存原文、标题、语言、字数、创建时间。
- Markdown 中的代码块允许存在，但不能直接原样进入字幕。

---

### 功能 2：文章解析

**功能描述**
- 系统基于原文生成结构化摘要，供分镜生成使用。

**Input**
- `project_id`
- 原文内容

**Output**
```json
{
  "topic": "RAG 基础概念",
  "audience": "对大模型应用有基础认知的技术读者",
  "core_message": "RAG 的核心价值在于解决知识更新和引用可信问题",
  "key_points": ["...", "..."],
  "visualizable_points": ["...", "..."],
  "discarded_parts": ["...", "..."],
  "content_type": "tutorial",
  "confidence": 0.86
}
```

**行为规则**
- 使用 LLM 输出结构化 JSON。
- 必须经过 JSON Schema 校验。
- 校验失败时自动重试一次。
- 若两次均失败，任务进入 `failed`，返回人工可读错误。
- `confidence` 低于阈值时，提示用户文章结构不清晰，不自动继续生成。
- 输出内容必须仅基于原文，不允许补充原文中不存在的新观点。

---

### 功能 3：分镜生成

**功能描述**
- 系统基于文章解析结果生成 `SceneSpec[]`，供模板和渲染模块使用。

**Input**
- `project_id`
- `ArticleAnalysis`
- 目标时长：45 到 60 秒

**Output**
```json
[
  {
    "scene_id": "sc_001",
    "order": 1,
    "template_type": "hook_title",
    "goal": "用反常识观点建立开头钩子",
    "voiceover": "你以为 RAG 只是给大模型接一个知识库？其实它真正解决的是知识更新和可信引用问题。",
    "screen_text": ["RAG 不只是知识库", "它解决知识更新与引用可信"],
    "duration_sec": 6,
    "pace": "fast",
    "transition": "cut",
    "visual_params": {
      "emphasis": "RAG"
    }
  }
]
```

**行为规则**
- Scene 数固定在 6 到 10 个。
- 每个 Scene 必须包含：
  - `scene_id`
  - `order`
  - `template_type`
  - `goal`
  - `voiceover`
  - `screen_text`
  - `duration_sec`
  - `pace`
  - `transition`
  - `visual_params`
- 允许的模板类型只有：
  - `hook_title`
  - `bullet_explain`
  - `compare_process`
- 单个 Scene 建议时长 4 到 9 秒。
- 单个 Scene 旁白最长 90 个中文字符。
- `screen_text` 最多 3 条，每条最长 18 个中文字符。
- 分镜生成后必须经过规则校验。
- 校验失败时允许 LLM 修复一次。
- 再失败则任务终止，不进入渲染。

---

### 功能 4：模板映射与模板渲染

**功能描述**
- 系统将 `SceneSpec` 映射为模板参数，再交给 Remotion 模板渲染。

**Input**
- `SceneSpec[]`

**Output**
- 可用于渲染的视频 props
- 每个 Scene 对应模板参数对象

**行为规则**
- 本版本仅支持 3 个模板：
  - `hook_title`：开头钩子、反常识观点、标题冲击
  - `bullet_explain`：要点解释、列表展开
  - `compare_process`：对比、流程、步骤说明
- LLM 不允许输出可执行代码，只允许输出 JSON 参数。
- 每个模板必须有独立参数 Schema。
- 参数缺失时先尝试补全；无法补全则拒绝渲染。
- 模板必须支持：
  - 9:16 竖屏
  - 字幕安全区
  - 不同长度文案的基础适配

---

### 功能 5：TTS 配音

**功能描述**
- 系统根据每个 Scene 的 `voiceover` 生成音频。

**Input**
```json
{
  "provider": "azure",
  "voice_id": "zh-CN-XiaoxiaoNeural",
  "speed": 1.08
}
```

**Output**
- 每个 Scene 的音频文件 URL
- 音频时长
- 如供应商提供则返回时间戳

**行为规则**
- 本版本只接入 1 个 TTS 供应商。
- 音频生成失败时允许重试。
- 连续失败时任务终止，并记录错误码和错误信息。
- 实际 Scene 时长以 TTS 音频时长为准，不以 LLM 建议时长为准。
- TTS 输入文本必须与最终字幕文本保持一致来源。

---

### 功能 6：字幕生成

**功能描述**
- 系统基于 TTS 结果生成句级或短语级字幕。

**Input**
- 每个 Scene 的最终旁白文本
- TTS 时间戳或字符估算结果

**Output**
```json
[
  {
    "cue_id": "sub_001",
    "scene_id": "sc_001",
    "start_ms": 0,
    "end_ms": 1800,
    "text": "RAG 不只是知识库。"
  }
]
```

**行为规则**
- 优先使用 TTS 供应商返回的时间戳。
- 若无时间戳，则按标点切句并按字符数估算。
- 不做字级字幕，只做句级或短语级。
- 单次字幕展示不得超过两行。
- 字幕文本必须来源于最终 TTS 文本，而不是用户原始草稿。

---

### 功能 7：视频渲染与导出

**功能描述**
- 系统将 Scene、模板参数、字幕、配音统一渲染为 MP4。

**Input**
- `SceneSpec[]`
- 模板参数
- 音频资产
- 字幕资产
- 样式配置

**Output**
```json
{
  "video_url": "https://cdn.example.com/videos/proj_xxx/final.mp4",
  "subtitle_url": "https://cdn.example.com/videos/proj_xxx/subtitles.srt",
  "scene_json_url": "https://cdn.example.com/videos/proj_xxx/scenes.json",
  "duration_sec": 57,
  "resolution": "1080x1920"
}
```

**行为规则**
- 固定输出规格：
  - 分辨率 `1080x1920`
  - 纵横比 `9:16`
  - 帧率 `30fps`
  - 格式 `MP4`
- 渲染任务必须异步执行。
- API 启动生成时只返回 `job_id`。
- 前端通过轮询查询状态。
- Job 超时后必须标记失败，不能无限停留在 `running`。
- 不对外输出 Remotion 工程文件。

---

### 功能 8：Scene 编辑

**功能描述**
- 用户可以对已生成的 Scene 做基础修改。

**Input**
```json
{
  "voiceover": "很多人以为 RAG 只是知识库问答，但它真正解决的是知识更新和引用可信的问题。",
  "screen_text": ["RAG 不只是知识库问答", "它解决知识更新与引用可信"],
  "template_type": "bullet_explain",
  "duration_sec": 7,
  "version": 4
}
```

**Output**
```json
{
  "scene_id": "sc_001",
  "version": 5,
  "validation": {
    "ok": true,
    "warnings": []
  }
}
```

**行为规则**
- 用户仅可编辑：
  - `voiceover`
  - `screen_text`
  - `template_type`
  - `duration_sec`
- 用户不可直接编辑底层模板参数结构。
- 编辑后必须再次走后端规则校验。
- 若版本冲突，返回 `409 Conflict`。
- 若编辑内容破坏约束，例如旁白过长、屏幕文案过多，则拒绝保存。
- 本版本不承诺 Scene 局部重生成，只支持编辑后整片重渲染。

---

### 功能 9：任务状态与存储

**功能描述**
- 系统以异步任务方式运行生成流程，并保存中间产物。

**Input**
- 生成请求
- 重渲染请求

**Output**
```json
{
  "job_id": "job_xxx",
  "status": "running",
  "stage": "tts_generate",
  "progress": 0.64,
  "error": null
}
```

**行为规则**
- Job 状态枚举固定为：
  - `queued`
  - `running`
  - `waiting_retry`
  - `failed`
  - `completed`
  - `cancelled`
- Pipeline 阶段固定为：
  - `article_parse`
  - `scene_generate`
  - `scene_validate`
  - `tts_generate`
  - `subtitle_generate`
  - `render_prepare`
  - `remotion_render`
  - `export`
- 必须保存：
  - 原文
  - ArticleAnalysis
  - SceneSpec
  - LLM 输入输出
  - 校验错误
  - TTS 文件
  - 字幕文件
  - 渲染日志
- 当用户重复发起渲染时，旧任务结果必须失效或被取消，避免污染当前结果。

## 5. 用户流程（User Flow）

### 首次生成流程
1. 用户输入标题和正文，提交文章。
2. 系统校验正文长度、格式和语言。
3. 系统创建 `Project`。
4. 用户点击“开始生成”。
5. 系统创建 `GenerationJob`。
6. 系统执行文章解析。
7. 系统执行分镜生成。
8. 系统执行 Scene 规则校验。
9. 校验通过后，系统生成 TTS 音频。
10. 系统基于 TTS 结果生成字幕。
11. 系统将 Scene、字幕、音频和模板参数送入 Remotion。
12. 系统异步渲染 MP4。
13. 系统保存视频结果和中间产物。
14. 前端轮询到 `completed` 后展示视频预览和 Scene 列表。

### 编辑后重渲染流程
1. 用户在 Scene 列表中修改某个 Scene 的文案、模板类型或时长。
2. 系统校验修改内容。
3. 系统创建新 `scene_version`。
4. 若某个 Scene 的旁白未变化，可复用旧 TTS 资产。
5. 若旁白变化，则重新生成该 Scene 的 TTS。
6. 系统重新生成全片字幕时间轴。
7. 系统整片重新渲染 MP4。
8. 系统输出新版本视频，旧版本保留可回滚。

## 6. 关键决策说明

### 技术妥协

- 接受技术经理意见：P0 不支持文章链接输入。
  原因：该能力引入抓取和清洗复杂度，不服务当前核心验证。

- 接受技术经理意见：P0 模板从 5 到 8 个缩减为 3 个。
  原因：先验证成片质量，而不是模板数量。

- 接受技术经理意见：不做复杂 Visual DSL，只做轻量 `SceneSpec`。
  原因：当前只需稳定 schema 和模板映射，不需要通用抽象层。

- 接受技术经理意见：Scene 级局部重生成改为“Scene 编辑 + 整片重渲染”。
  原因：这能保留可编辑性，同时避免时间轴和缓存一致性复杂度。

- 接受技术经理意见：视频时长固定在 45 到 60 秒。
  原因：30 秒和 90 秒是不同节奏模型，本版本只做一个可控长度段。

- 接受程序员反馈：字幕只做到句级或短语级，不做字级精准同步。
  原因：当前收益不足以覆盖实现成本。

- 接受程序员反馈：只接 1 个 TTS 供应商，不抽象多供应商接口。
  原因：当前版本重在闭环，不重在扩展性。

### 产品坚持

- 拒绝“先按段落切视频即可”的简化建议。
  原因：段落分页不是视频化表达，会直接损害成片质量和付费意愿。

- 坚持保留文章解析。
  原因：没有结构化理解，后续分镜和模板适配质量不稳定。

- 坚持保留分镜生成。
  原因：这是本产品的核心价值，不可退化成字幕朗读工具。

- 坚持保留 3 个固定模板而不是 1 个模板。
  原因：只有 1 个模板会让所有视频看起来像同一种 PPT，不足以支撑解释、对比、钩子三类高频表达。

- 坚持保留 Scene 编辑能力。
  原因：生成式结果不可能一次命中，不可编辑的产品无法进入真实使用流程。

- 坚持保留 MP4 成片导出。
  原因：用户最终需要的是可发布视频，而不是 JSON 或工程文件。

## 7. 风险与边界

### 已知风险

- LLM 可能输出不合法 JSON。
  处理方式：Schema 校验 + 自动修复 + 限次重试。

- 文章结构差会导致分镜偏题。
  处理方式：输出 `confidence`，低置信度时中断生成并提示用户整理内容。

- TTS 时长会与 LLM 建议时长不一致。
  处理方式：最终渲染以 TTS 实际音频时长为准。

- 模板参数缺失可能导致渲染失败。
  处理方式：模板参数单独校验，缺字段时补全或拒绝渲染。

- Remotion 渲染耗时较长。
  处理方式：任务异步化，限制并发，设置超时。

- 用户连续点击生成会造成任务冲突。
  处理方式：旧任务取消或标记过期，只有最新有效任务可以更新当前结果。

- 字幕可能溢出或与语音不同步。
  处理方式：按标点切句，限制两行以内，字幕始终基于最终 TTS 文本生成。

- 中文字体缺失会造成显示异常。
  处理方式：渲染环境内置中文字体。

- 重复生成会放大成本。
  处理方式：记录 token、TTS 字数、渲染耗时，并限制项目生成次数。

### 明确不处理的情况

- 不处理 URL 抓取和网页正文抽取。
- 不处理 3000 字以上长文的自动拆分和多段合成。
- 不处理横屏视频。
- 不处理多平台适配。
- 不处理团队协作、权限、批量任务中心。
- 不处理品牌资产管理。
- 不处理素材混剪、图片检索和外部媒体拼接。
- 不处理字级字幕。
- 不处理开放式自由风格视频生成。
- 不处理用户直接编辑模板底层参数或动画轨道。
- 不保证所有类型文章都能转成高质量视频。
  当前仅支持解释型、教程型、对比型文章。

---

## 冻结结论

本版本正式冻结为以下一句话：

> 面向技术博主和知识创作者，用户输入一篇 1000 到 3000 字的 Markdown 或纯文本文章，系统生成 6 到 10 个 Scene，使用 3 个固定模板，自动生成配音、句级字幕和 45 到 60 秒的 9:16 MP4。用户可编辑 Scene 文案和模板类型，修改后整片重渲染。

这就是可直接进入开发的版本定义。
