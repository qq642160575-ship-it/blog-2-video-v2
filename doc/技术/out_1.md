# 1. 对 PRD 的总体评价

这个 PRD 的方向是对的，但当前写法有明显问题：它把“核心链路验证”和“可商业化生产系统”混在一起了。

最大的问题不是技术不可行，而是 MVP 里塞进了太多“正确但昂贵”的东西：Visual DSL、5 到 8 种视觉模板、Scene 级编辑、局部重生成、TTS、字幕对齐、Remotion 渲染、视频导出、文章解析、分镜生成。这不是一个轻 MVP，而是一个缩小版生产系统。

产品经理的问题是：把“最终产品应该具备的能力”直接塞进了 MVP。
程序员也要警惕：不要借 Visual DSL、模板引擎、任务系统、可编辑工程这些概念搞复杂架构炫技。第一阶段目标应该是验证“文章到短视频分镜 + 固定模板渲染”是否真的能被用户接受，而不是一上来搭一个内容生产平台。

我的判断：

- 技术可行，但端到端稳定交付难度高。
- MVP 范围过大，容易 2 到 3 个月做出一个半成品。
- 核心风险不是渲染，而是内容改写质量、分镜质量和模板适配质量。
- 真正应该先验证的是：用户是否愿意接受“有限模板 + 可编辑脚本 + 可导出视频”的产物。

# 2. 不合理需求列表

## 不合理点 1：MVP 同时要求 5 到 8 种视觉模板

为什么不该这么做：
5 到 8 种模板不是简单数量问题，每一种模板都意味着输入字段、动画规则、边界状态、字幕适配、时长适配、错误兜底和视觉一致性。模板越多，质量越难控。MVP 阶段最怕的不是模板少，而是每个模板都半成品。

推荐：

- 先做 3 种核心模板即可：
- 标题/观点冲击
- 列表/要点解释
- 对比/流程说明

代码解释、UI 演示框、时间线、数字强调可以先合并进通用模板，不要一开始独立设计。

## 不合理点 2：MVP 要做 Scene 级编辑与重生成

为什么不该这么做：
Scene 级编辑是对的，但“Scene 级重生成”容易过早引入复杂状态管理。比如用户改了第 3 个 Scene 的时长，会影响后续字幕时间轴、配音节奏、总时长、转场、背景音乐、导出缓存。如果没有稳定的数据模型和渲染缓存，局部重生成会变成大量隐性 bug。

推荐：

- MVP 做“Scene 级编辑”，但不要做真正复杂的“局部重生成”。
- 用户可以编辑 Scene 文案、视觉类型、屏幕文案。
- 保存后重新生成整条视频。
- 后台可以按 Scene 做缓存，但产品层不要承诺复杂的局部重生成体验。

## 不合理点 3：文章链接输入放进 P0

为什么不该这么做：
文章链接输入会引入网页抓取、反爬、正文抽取、登录墙、广告清洗、图片处理、编码异常、版权风险。这个能力和核心视频生成链路没有强绑定，却会显著拖慢 MVP。

推荐：

- P0 只支持纯文本粘贴和 Markdown 输入。
- 文章链接输入放到 P1，等核心生成链路稳定后再做。

## 不合理点 4：P0 要“输出可渲染的视频工程与成片”

为什么不该这么做：
“工程文件”和“成片”是两类用户价值。目标用户真正要的是 MP4 成片，不一定需要工程。输出工程会把产品推向专业剪辑/开发工具方向，增加解释成本和兼容成本。

推荐：

- P0 只承诺 MP4 成片和结构化 JSON 工程数据。
- 不要暴露 Remotion 工程文件给用户。内部可以用 Remotion，但不要把内部实现变成产品承诺。

## 不合理点 5：质量预警放在 P1，但其实应该拆成最小规则进入 P0

为什么不该这么做：
如果没有质量约束，LLM 很容易生成过长 Scene、过密字幕、节奏失控的脚本。后面模板和渲染做得再好，成片也会难看。质量预警不应该作为高级功能，而应该是生成管线里的硬规则。

推荐：

- P0 不做复杂评分系统，但必须有硬性校验：
- 单个 Scene 旁白不超过指定字数。
- 总 Scene 数控制在 6 到 10 个。
- 总时长控制在 30 到 60 秒，先不要做 90 秒。
- 每屏字幕不超过两行。
- 每个 Scene 必须有视觉类型。

## 不合理点 6：MVP 定义为 30 到 90 秒，范围太宽

为什么不该这么做：
30 秒和 90 秒是两种产品。30 秒强调高压缩、强钩子、节奏快；90 秒强调解释完整、信息密度和过渡。一个 MVP 同时支持这两个范围，会导致分镜规则、模板节奏、配音速度都变复杂。

推荐：

- MVP 固定做 45 到 60 秒。
- 先把一个长度段打磨稳定，再扩展到 30 秒快剪版和 90 秒解释版。

## 不合理点 7：批量处理被反复强调，但和 MVP 用户定义冲突

为什么不该这么做：
PRD 说 MVP 优先服务技术博主和知识博主，又强调批量能力是商业化信号。这是两个不同阶段。批量处理需要任务队列、失败重试、并发控制、成本预算、结果管理，这些不是核心体验验证必须项。

推荐：

- MVP 不做批量 UI，不做复杂任务中心。
- 但数据结构上保留 `job` 和 `project` 概念，后续可扩展。第一版只支持单任务生成。

## 不合理点 8：Visual DSL 被描述得过重

为什么不该这么做：
PRD 对 DSL 的判断有道理，但容易诱导技术团队做一套过度抽象的视觉语言系统。MVP 不需要“通用视觉协议”，只需要一个稳定的 Scene Schema 和模板参数映射。过早设计 DSL 会消耗大量时间在抽象上，而不是打磨真实成片质量。

推荐：

- 第一版不要叫 Visual DSL，先叫 `SceneSpec`。
- 字段保持简单：

```json
{
  "scene_id": "s1",
  "type": "hook | list | compare | process",
  "voiceover": "...",
  "screen_text": ["...", "..."],
  "duration_sec": 6,
  "visual_params": {},
  "transition": "cut | slide"
}
```

等模板数量超过 8 个、客户定制需求出现后，再升级为正式 DSL。

## 不合理点 9：目标用户太多，导致需求摇摆

为什么不该这么做：
技术博主、知识创作者、小型团队、AI 内容创业者的诉求不一样。个人创作者要省时间，小团队要流程和一致性，创业者要成本和规模化。PRD 虽然说 MVP 优先技术/知识博主，但后文又不断为批量、品牌、ROI、数据看板埋需求，容易导致产品方向失焦。

推荐：

- MVP 只服务一个主用户：技术/知识博主。
- 成功标准也只保留一个：输入一篇结构清晰文章，10 分钟内生成一条可发布初版视频。

# 3. 系统架构设计

建议高层架构不要一开始做成复杂平台，而是做成一条可观测的生成流水线。

核心链路：

```text
文章输入
  ↓
文本清洗与结构化
  ↓
内容摘要与短视频脚本生成
  ↓
SceneSpec 生成与校验
  ↓
模板选择与参数映射
  ↓
TTS 生成
  ↓
字幕时间轴生成
  ↓
Remotion 渲染
  ↓
MP4 导出
  ↓
用户预览与编辑
```

系统模块：

```text
Frontend Editor
  - 文章输入
  - Scene 列表编辑
  - 视频预览
  - 导出状态展示

Backend API
  - 项目管理
  - 生成任务管理
  - SceneSpec 存储
  - 渲染任务调度

LLM Pipeline
  - 文章摘要
  - 分镜生成
  - SceneSpec 校验修复

Template Renderer
  - 固定模板组件
  - 参数映射
  - Remotion composition

Media Pipeline
  - TTS
  - 字幕对齐
  - 音频合成
  - MP4 渲染

Storage
  - 原文
  - SceneSpec
  - 音频文件
  - 渲染结果
  - 任务日志
```

架构原则：

- 不允许 LLM 直接生成可执行代码。
- 不要让前端承担生成逻辑。
- 不要一开始做多租户复杂权限。
- 不要一开始做复杂工作流编排。
- 所有生成步骤必须保存中间产物，否则无法调试质量问题。

# 4. 技术模块拆解

## 文章输入模块

- P0 只支持纯文本和 Markdown。
- 不做 URL 抓取。
- 需要做最基本的长度校验、空内容校验、段落切分。

## 内容理解模块

- 输入文章，输出主题、受众、核心观点、可视频化要点。
- 不要追求复杂 NLP 系统，直接用 LLM + JSON Schema。
- 必须有失败重试和 JSON 修复。

## 分镜生成模块

- 输出 SceneSpec。
- 每个 Scene 必须包含旁白、屏幕文案、视觉类型、建议时长。
- 需要硬性校验，不合格就让模型重写。

## 模板模块

- 第一版 3 个核心模板。
- 模板参数必须稳定，不允许模型自由发挥 CSS 或 React 代码。
- 所有模板要支持 9:16、字幕安全区和不同文案长度。

## TTS 与字幕模块

- TTS 先选一个供应商，不要抽象多供应商适配层。
- 字幕先用文本估算或 TTS 返回时间戳。
- 不要一开始追求字级精准，句级/短语级足够 MVP。

## 渲染模块

- Remotion 可行。
- 渲染服务要异步任务化。
- P0 不做高并发，只要支持单用户等待和失败提示。

## 编辑模块

- 允许编辑 Scene 文案、屏幕文字、视觉类型。
- 不做复杂时间轴编辑器。
- 不做拖拽动画配置。
- 修改后重新生成整片。

## 任务与存储模块

- 保留 `project`、`generation_job`、`scene`、`asset` 四类数据。
- 不做复杂批量任务中心。
- 必须保存每次生成输入和输出，方便排查生成质量。

# 5. 替代方案

## 推荐 MVP 方案：更简单、更稳

第一版目标改成：

> 输入一篇 1000 到 3000 字文章，生成一条 45 到 60 秒、9:16、带字幕和配音的动态讲解 MP4，用户可编辑分镜文案后整片重渲染。

范围调整：

- 保留：文章解析、分镜生成、3 个模板、TTS、字幕、MP4 导出、Scene 文案编辑。
- 删除：文章链接输入、5 到 8 种模板、复杂 Visual DSL、局部重生成、批量处理、多平台、多风格、品牌系统。
- 延后：质量预警 UI、A/B 开头、封面标题、数据看板、素材混剪。

更稳的实现路径：

1. 先做“文本到 SceneSpec”的离线脚本，验证 20 篇文章生成质量。
2. 再做 3 个 Remotion 模板，把 SceneSpec 渲染成视频。
3. 再接 TTS 和字幕。
4. 最后做 Web UI 和编辑能力。
5. 等 20 到 50 条样片稳定后，再扩展模板数量。

这是比 PRD 更合理的路线。否则团队会一边做 UI、一边做 LLM、一边做模板、一边做渲染、一边做编辑，最后每块都不稳定。

# 6. 风险点

## 性能风险

- Remotion 渲染可能耗 CPU，视频生成时间不可忽视。
- TTS 生成和视频渲染都属于耗时任务，不能同步阻塞请求。
- 多用户并发时，渲染队列会成为瓶颈。

建议：
P0 使用异步任务队列，限制并发。不要一开始追求实时生成。

## 扩展风险

- 如果 SceneSpec 设计太随意，后面模板扩展会痛苦。
- 如果模板参数不稳定，LLM 每次输出都要人工兜底。
- 如果早期让模型生成代码，后面会很难维护。

建议：
固定 Schema，固定模板输入，不让 LLM 越权。

## 成本风险

- LLM 调用、TTS、视频渲染都会产生单位成本。
- 90 秒视频比 45 秒视频成本明显更高。
- 批量处理会放大失败重试成本。

建议：
MVP 先限制长度、Scene 数、生成次数。每次生成都记录 token、TTS 字数、渲染耗时。

## 质量风险

- 文章质量差会直接导致视频质量差。
- 分镜生成不稳定会让视频像 PPT。
- 模板过少会让所有视频长得一样。
- 模板过多又会导致实现失控。

建议：
先限定文章类型，只支持解释型、教程型、对比型。不要承诺所有文章都能转。

## 产品风险

- 用户可能不需要“完全自动生成”，而是需要“快速生成可改初稿”。
- 如果宣传成一键爆款视频，交付一定翻车。
- 如果编辑能力太弱，用户无法进入真实工作流。

建议：
产品定位改成“文章转短视频初稿工具”，不要包装成“全自动短视频生产系统”。

# 7. 【关键】否定产品的点 + 推荐方案

## 否定点 1：否定当前 MVP 范围

为什么不该这么做：
当前 MVP 实际包含了完整产品的核心基础设施，范围过大，交付风险高。

推荐方案：
MVP 收敛为“纯文本输入 + SceneSpec + 3 模板 + TTS + 字幕 + MP4 + 简单编辑”。

## 否定点 2：否定 P0 支持文章链接

为什么不该这么做：
URL 抓取是旁路复杂度，不验证核心价值。

推荐方案：
P0 只支持粘贴正文和 Markdown。

## 否定点 3：否定 5 到 8 个模板作为 MVP 必须项

为什么不该这么做：
模板数量会直接拉高设计、开发、测试、适配成本。

推荐方案：
先做 3 个高频模板，把成片质量打磨到可发布，再扩展。

## 否定点 4：否定复杂 Visual DSL

为什么不该这么做：
过早 DSL 化会诱导技术团队做抽象系统，而不是验证视频质量。

推荐方案：
先做轻量 `SceneSpec`，只服务当前 3 到 5 类模板。等业务稳定后再演进为 DSL。

## 否定点 5：否定 Scene 级局部重生成作为 P0

为什么不该这么做：
局部重生成会引入时间轴、音频、字幕、缓存一致性问题，复杂度高于表面描述。

推荐方案：
P0 只做 Scene 级编辑，编辑后整片重渲染。内部可预留缓存，不对用户承诺局部重生成。

## 否定点 6：否定批量处理进入 MVP

为什么不该这么做：
批量处理不是简单循环生成，而是任务系统、失败恢复、成本控制和结果管理。

推荐方案：
P0 数据模型预留任务概念，但产品界面只做单条生成。

## 否定点 7：否定“一开始支持 30 到 90 秒”

为什么不该这么做：
时长跨度太大，会影响脚本生成、Scene 数、配音节奏和模板表现。

推荐方案：
P0 固定 45 到 60 秒。后续再做 30 秒精简版和 90 秒详细版。

## 最终推荐版本

我建议把 MVP 改成：

> 面向技术/知识博主，支持粘贴一篇结构清晰的文章，系统生成 6 到 10 个 Scene，使用 3 种固定视觉模板，自动生成 TTS、字幕和 9:16 MP4。用户可以编辑每个 Scene 的文案和视觉类型，修改后整片重渲染。目标是在 10 分钟内拿到一条 45 到 60 秒的可发布初版。

这是更合理的技术边界。
它保留了产品核心价值：文章视频化。
它砍掉了不必要复杂度：链接抓取、复杂 DSL、批量、局部重生成、多模板膨胀。
它也能防止程序员过度工程化：先把端到端质量跑通，再谈平台化。
