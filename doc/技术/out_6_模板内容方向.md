# 技术评审报告：Scene 表达力与节奏数据优化方案

## 🎯 总体评价

**这个 PRD 有严重的过度设计问题。**

产品经理把一个"视频节奏优化"的需求，包装成了"三层数据架构"，但没有证明这个复杂度是必要的。更关键的是：**你们还没验证过用户真的需要这些功能**。

---

## ❌ 不合理需求列表

### 1. **表达层数据结构过度复杂**
**问题**：
- `emphasis_words`、`pause_points`、`tone_shifts`、`speech_rate_changes` 这 4 个字段，真的都需要吗？
- 你们有数据证明"语气变化"比"停顿位置"更重要吗？
- 为什么不先做 1-2 个最关键的字段，验证效果后再扩展？

**为什么不该这么做**：
- LLM 输出字段越多，格式错误率越高
- 你们现在 95% 成功率，加了这么多字段后可能掉到 70%
- 调试成本会爆炸：哪个字段出错了？是 LLM 的问题还是 Prompt 的问题？

### 2. **时间轴数据结构是"伪需求"**
**问题**：
- `visual_keyframes` 要求 LLM 输出精确到 0.1 秒的时间点
- 但 LLM 根本不知道 TTS 会生成多长的音频
- 你们打算让 LLM "猜" 时间轴，然后再用"线性缩放"修正？

**为什么不该这么做**：
- 这是典型的"先设计数据结构，再想怎么填数据"
- 正确做法：先生成 TTS，再根据实际音频时长计算时间轴
- 你们这个方案是"先猜时间轴，再缩放"，误差会累积

### 3. **LLM Prompt 要求不现实**
**问题**：
```
- 关键词必须在说到时同步出现（误差 < 0.2 秒）
- 每个视觉元素出现时间 = 对应音频时间 - 0.1 秒（预判）
```

**为什么不该这么做**：
- LLM 不是时间轴编辑器，它不知道"2.5 秒"是什么概念
- 你们要求 LLM 输出"误差 < 0.2 秒"的时间点，但 LLM 连 TTS 时长都不知道
- 这个需求本质上是"让 LLM 做它做不到的事"

### 4. **验收标准不可测**
**问题**：
- "重音词准确率 ≥ 80%（人工评估）" —— 谁来评？标准是什么？
- "节奏起伏明显率 ≥ 70%（自动检测：标准差 > 0.5）" —— 标准差是什么的标准差？

**为什么不该这么做**：
- 人工评估成本高，且主观性强
- "标准差 > 0.5" 这种指标没有业务意义
- 你们应该测"用户修改率是否下降"，而不是这些中间指标

---

## 🏗️ 系统架构设计（高层）

如果真要做这个需求，架构应该是：

```
1. 内容生成层
   - LLM 生成 voiceover + screen_text（现有）
   - LLM 标注"重点词"（新增，最小化）

2. TTS 生成层
   - 调用 TTS API 生成音频
   - 获取音频时长和词级时间戳（word-level timestamps）

3. 时间轴计算层（新增）
   - 根据 TTS 返回的时间戳，计算视觉元素出现时机
   - 根据"重点词"，生成强调动效的时间点

4. 渲染层
   - 读取时间轴数据，渲染视频（现有，微调）
```

**关键点**：
- 时间轴数据不是 LLM 生成的，是根据 TTS 结果计算的
- LLM 只需要标注"哪些词重要"，不需要标注"什么时候出现"

---

## 🔧 技术模块拆解

### 模块 1：LLM 输出扩展（2 天）
**做什么**：
- Scene 增加 `emphasis_words: List[str]` 字段
- Prompt 增加"标注 2-3 个关键词"的要求

**不做什么**：
- ❌ 不做 `pause_points`（TTS 自己会处理停顿）
- ❌ 不做 `tone_shifts`（TTS 不支持）
- ❌ 不做 `speech_rate_changes`（过度设计）

### 模块 2：TTS 时间戳提取（1 天）
**做什么**：
- 调用 TTS API 时，获取 word-level timestamps
- 存储到 Scene 的 `tts_metadata` 字段

**技术风险**：
- 你们用的 TTS 服务支持 word-level timestamps 吗？
- 如果不支持，这个方案就做不了

### 模块 3：时间轴计算服务（3 天）
**做什么**：
- 根据 TTS timestamps，计算每个 `screen_text` 的出现时机
- 根据 `emphasis_words`，生成强调动效的时间点

**算法**：
```python
def calculate_timeline(scene, tts_metadata):
    timeline = []
    for word in scene.emphasis_words:
        # 找到这个词在 TTS 中的时间戳
        timestamp = find_word_timestamp(word, tts_metadata)
        # 提前 0.1 秒显示
        timeline.append({
            'time': timestamp - 0.1,
            'element': word,
            'action': 'pop'
        })
    return timeline
```

### 模块 4：渲染层适配（2 天）
**做什么**：
- 读取时间轴数据
- 在指定时间点触发动效

**不做什么**：
- ❌ 不做复杂动效（只做 fade_in, pop, slide_in）
- ❌ 不做音频同步点（TTS timestamps 已经是同步点）

**总计**：8 天，约 400 行代码（不是 600 行）

## 💡 替代方案（更简单 or 更稳）

### 方案 A：最小化方案（推荐）
**做什么**：
1. LLM 只标注"重点词"（2-3 个）
2. TTS 返回 word-level timestamps
3. 根据 timestamps，自动在重点词出现时加强调动效

**优点**：
- LLM 输出简单，成功率高
- 不需要"猜"时间轴，直接用 TTS 数据
- 开发成本低（5 天）

**缺点**：
- 只能做"重点词强调"，不能做"节奏变化"

### 方案 B：规则生成方案
**做什么**：
1. 不改 LLM 输出
2. 根据 Scene 类型（Hook/解释/对比），应用预设节奏规则
3. 例如：Hook 场景自动加快语速，解释场景自动放慢

**优点**：
- 不依赖 LLM，稳定性高
- 开发成本极低（3 天）

**缺点**：
- 不够灵活，无法适应特殊内容

### 方案 C：后处理方案
**做什么**：
1. 先生成视频（现有流程）
2. 用户预览后，可以手动调整节奏
3. 前端提供简单的"加快/放慢/强调"按钮

**优点**：
- 不改后端，风险低
- 用户可以精确控制

**缺点**：
- 需要用户手动操作

---

## ⚠️ 风险点

### 1. **性能风险**
- 你们说"渲染时长 < 10 秒"，但加了时间轴计算后，可能会变成 15 秒
- 关键帧数量限制在 10 个，但一个 60 秒的视频，10 个关键帧够吗？

### 2. **扩展性风险**
- 你们设计的数据结构是"写死"的（fade_in, pop, slide_in）
- 如果以后要加新动效，要改 Schema 吗？
- 建议：动效类型应该是可配置的，不是硬编码的

### 3. **成本风险**
- 你们说"成本增加 15%"，但这是基于"LLM 输出成功率不变"的假设
- 如果成功率从 95% 降到 80%，重试成本会让总成本增加 50%+

### 4. **依赖风险**
- 整个方案依赖 TTS 服务支持 word-level timestamps
- 如果 TTS 服务不支持，或者精度不够，方案就做不了
- **你们验证过 TTS 服务的能力了吗？**

## ✅ 我的推荐方案

### 第一阶段（MVP，5 天）
1. LLM 标注"重点词"（2-3 个）
2. TTS 返回 word-level timestamps
3. 根据 timestamps，在重点词出现时加强调动效（pop）

**验收标准**：
- 用户认为"关键信息被强调"的比例 ≥ 60%
- 用户修改率下降 ≥ 5%

### 第二阶段（如果 MVP 成功，3 天）
1. 根据 Scene 类型，应用预设节奏规则
2. Hook 场景：快速 + 停顿
3. 解释场景：中速 + 均匀

**验收标准**：
- 用户认为"节奏有起伏"的比例 ≥ 50%
- 用户修改率下降 ≥ 10%

### 第三阶段（如果前两阶段成功，5 天）
1. 前端提供简单的节奏编辑器
2. 用户可以手动调整"加快/放慢/强调"

**验收标准**：
- 使用编辑器的用户比例 ≥ 20%
- 使用编辑器后的满意度 ≥ 4.5 分

**不要做产品提出的方案。**

原因：
1. 过度设计，风险高
2. 技术路线错误（让 LLM 生成时间轴）
3. 验收标准不可测
4. 没有验证用户需求

**应该做我推荐的方案。**

原因：
1. 最小化设计，风险低
2. 技术路线正确（先 TTS，再计算时间轴）
3. 验收标准可测（用户修改率）
4. 分阶段验证，快速迭代
