## 0. 技术评审反馈总结

技术团队指出了 v1 方案的 4 个核心问题：
1. **过度设计**：表达层 4 个字段太多，LLM 输出成功率会下降
2. **技术路线错误**：让 LLM 生成时间轴是错的，应该先 TTS 再计算
3. **验收标准不可测**：人工评估成本高，应该测用户修改率
4. **没有验证假设**：缺少用户调研数据支撑

**我接受技术的核心观点，但坚持产品价值判断。**

## 1. 产品立场声明

### 1.1 我接受的技术观点

✅ **接受**：让 LLM 生成精确时间轴是错的
- 技术说得对，LLM 不知道 TTS 时长，让它"猜"时间点会累积误差
- 正确做法：先 TTS，再根据 word-level timestamps 计算时间轴

✅ **接受**：表达层字段应该最小化
- 4 个字段确实太多，应该先做 1-2 个最关键的
- `emphasis_words` 是核心，其他字段可以后续验证后再加

✅ **接受**：验收标准应该可测
- "重音词准确率"确实难测，应该改为"用户修改率下降 X%"
- 业务指标比技术指标更有意义

### 1.2 我不接受的技术观点

❌ **不接受**："这是过度设计，不该做"
- 技术认为这是过度设计，但我认为这是**必要复杂度**
- 当前系统的核心问题就是"内容对但不抓人"，不解决表达和节奏问题，产品永远是"效率工具"

❌ **不接受**："先做规则方案，不要动 LLM"
- 规则方案（方案 B）确实成本低，但效果上限也低
- Hook 场景自动加快、解释场景自动放慢，这种"一刀切"无法适应内容差异
- 我们的目标是"像人说话"，不是"按类型套模板"

❌ **不接受**："用户需求没验证，不该做"
- 技术质疑"用户修改率 30%，其中 60% 是因为节奏不对"的数据来源
- 我承认这个数据是推测，但**不做用户调研就不能做产品是错的**
- 正确做法：快速做 MVP 验证，而不是先花 2 周做调研

### 1.3 我的妥协方案

我接受技术的核心建议，但坚持产品方向：

**妥协点 1**：采用技术推荐的"最小化方案"作为 MVP
- 只做 `emphasis_words` 标注
- 时间轴由 TTS timestamps 计算，不让 LLM 生成
- 开发周期从 8 天降到 5 天

**妥协点 2**：验收标准改为可测的业务指标
- 不测"重音词准确率"，测"用户修改率下降 5%"
- 不测"节奏起伏标准差"，测"用户认为关键信息被强调的比例 ≥ 60%"

**坚持点 1**：必须做 LLM 标注，不能只做规则
- 规则方案成本低但效果差，无法适应内容差异
- LLM 标注是产品差异化的关键

**坚持点 2**：必须分阶段验证，但不能因为"没数据"就不做
- MVP 验证"重点词强调"是否有效
- 如果有效，再做"节奏规则"
- 如果仍有效，再做"可视化编辑器"

## 2. 核心问题定义（不变）

### 问题 1：Scene 不够"可表达"
当前 Scene 只能控制"说什么"，无法控制"怎么说"。

### 问题 2：没有"节奏数据"
当前只有总时长和快中慢，没有细粒度的时间轴数据。

## 3. 调整后的方案

### 3.1 技术架构（采用技术建议）

```
1. 内容生成层
   - LLM 生成 voiceover + screen_text（现有）
   - LLM 标注 emphasis_words（新增，2-3 个关键词）

2. TTS 生成层
   - 调用 TTS API 生成音频
   - 获取 word-level timestamps（新增）

3. 时间轴计算层（新增）
   - 根据 TTS timestamps，计算视觉元素出现时机
   - 根据 emphasis_words，生成强调动效的时间点

4. 渲染层
   - 读取时间轴数据，在指定时间点触发动效（微调）
```

**关键变化**：
- ✅ 时间轴不由 LLM 生成，而是根据 TTS 结果计算
- ✅ LLM 只标注"哪些词重要"，不标注"什么时候出现"
- ✅ 表达层字段从 4 个减少到 1 个（`emphasis_words`）

### 3.2 数据结构设计

**Scene 新增字段**：
```python
# 表达层（最小化）
emphasis_words: List[str] = Field(
    description="需要强调的关键词，2-3 个",
    min_items=2,
    max_items=3
)

# TTS 元数据（新增）
tts_metadata: dict = Field(
    description="TTS 返回的 word-level timestamps",
    default=None
)

# 时间轴数据（计算生成，不由 LLM 输出）
timeline_data: dict = Field(
    description="根据 TTS timestamps 计算的时间轴",
    default=None
)
```

**为什么只要这 3 个字段**：
- `emphasis_words`：LLM 标注，告诉系统"哪些词重要"
- `tts_metadata`：TTS 返回，提供精确的时间戳
- `timeline_data`：计算生成，不需要 LLM 输出

### 3.3 LLM Prompt 设计（简化）

```
你是短视频脚本专家。基于文章分析和 Hook，生成 6-10 个场景。

【核心要求】
每个场景必须包含：
1. voiceover: 旁白文本（口语化、简洁）
2. screen_text: 屏幕文字（2-4 个关键词）
3. emphasis_words: 需要强调的关键词（2-3 个）

【emphasis_words 标注规则】
- 从 voiceover 中选择最重要的 2-3 个词
- 这些词应该是：核心概念、数字、对比词、结论词
- 例如：voiceover = "这个方法能让你的效率提升 3 倍"
  emphasis_words = ["方法", "效率", "3 倍"]

【输出格式】
JSON，每个场景包含 voiceover, screen_text, emphasis_words
```

**关键变化**：
- ✅ 不要求 LLM 输出时间点
- ✅ 不要求 LLM 输出停顿、语气、语速
- ✅ 只要求标注 2-3 个关键词

### 3.4 时间轴计算算法

```python
def calculate_timeline(scene, tts_metadata):
    """根据 TTS timestamps 计算时间轴"""
    timeline = []

    # 遍历每个需要强调的词
    for word in scene.emphasis_words:
        # 在 TTS metadata 中找到这个词的时间戳
        timestamp = find_word_timestamp(word, tts_metadata)

        if timestamp:
            # 提前 0.1 秒显示（预判）
            timeline.append({
                'time': timestamp - 0.1,
                'element': word,
                'action': 'pop',  # 强调动效
                'duration': 0.3
            })

    # 为 screen_text 生成出现时机
    for i, text in enumerate(scene.screen_text):
        # 均匀分布在场景时长内
        time = (i + 1) * scene.duration_sec / (len(scene.screen_text) + 1)
        timeline.append({
            'time': time,
            'element': text,
            'action': 'fade_in',
            'duration': 0.4
        })

    return timeline
```

**关键点**：
- 时间轴是计算出来的，不是 LLM 生成的
- 依赖 TTS 的 word-level timestamps
- 算法简单，易于调试

## 4. 分阶段实施计划

### 第一阶段：MVP（5 天）

**目标**：验证"重点词强调"是否有效

**做什么**：
1. Scene 增加 `emphasis_words` 字段
2. LLM Prompt 增加"标注 2-3 个关键词"
3. TTS 调用时获取 word-level timestamps
4. 实现时间轴计算算法
5. 渲染层在重点词出现时加 pop 动效

**不做什么**：
- ❌ 不做停顿、语气、语速控制
- ❌ 不做复杂动效（只做 pop）
- ❌ 不做可视化编辑器

**验收标准**：
- LLM 输出成功率 ≥ 90%（允许从 95% 下降 5%）
- 用户认为"关键信息被强调"的比例 ≥ 60%
- 用户修改率下降 ≥ 5%（从 30% 降到 25%）

**如果失败**：
- 如果成功率 < 90%，说明 LLM 标注不稳定，回退到规则方案
- 如果用户感知 < 60%，说明 pop 动效不够明显，调整动效参数
- 如果修改率没下降，说明"重点词强调"不是用户痛点，停止开发

### 第二阶段：节奏规则（3 天）

**前提**：第一阶段验收通过

**目标**：验证"节奏起伏"是否有效

**做什么**：
1. 根据 Scene 类型（Hook/解释/对比），应用预设节奏规则
2. Hook 场景：关键词密集出现 + 快速动效
3. 解释场景：关键词均匀分布 + 中速动效
4. 对比场景：关键词成对出现 + 交替动效

**验收标准**：
- 用户认为"节奏有起伏"的比例 ≥ 50%
- 用户修改率下降 ≥ 10%（从 30% 降到 20%）

**如果失败**：
- 如果用户感知 < 50%，说明规则不够灵活，需要 LLM 生成节奏数据
- 如果修改率没下降，说明节奏不是主要问题，停止开发

### 第三阶段：可视化编辑器（5 天）

**前提**：第二阶段验收通过

**目标**：让用户可以微调节奏

**做什么**：
1. 前端提供时间轴编辑器
2. 用户可以拖动关键帧调整时机
3. 用户可以点击词语标记重音
4. 用户可以调整动效类型（pop/fade_in/slide_in）

**验收标准**：
- 使用编辑器的用户比例 ≥ 20%
- 使用编辑器后的满意度 ≥ 4.5 分

## 5. 技术实现细节

### 5.1 模块拆解

**模块 1：LLM 输出扩展（2 天）**
- Scene 模型增加 `emphasis_words` 字段
- SceneGenerateService 的 Prompt 增加标注要求
- 解析 LLM 输出，提取 `emphasis_words`
- 单元测试

**模块 2：TTS 时间戳提取（1 天）**
- 调用 TTS API 时，请求 word-level timestamps
- 存储到 Scene 的 `tts_metadata` 字段
- 处理 TTS 不支持 timestamps 的降级逻辑

**模块 3：时间轴计算服务（1 天）**
- 实现 `calculate_timeline()` 函数
- 根据 `emphasis_words` 和 `tts_metadata` 计算时间轴
- 存储到 Scene 的 `timeline_data` 字段

**模块 4：渲染层适配（1 天）**
- 读取 `timeline_data`
- 在指定时间点触发 pop 动效
- 测试音频视觉同步精度

**总计**：5 天，约 300 行代码

### 5.2 关键技术风险

**风险 1：TTS 服务不支持 word-level timestamps**
- **验证方法**：立即测试当前 TTS 服务是否支持
- **应对方案**：如果不支持，使用 forced alignment 工具（如 Gentle）
- **降级方案**：如果都不行，用规则估算时间点（按字数均匀分布）

**风险 2：LLM 标注的词在 TTS 中找不到**
- **原因**：LLM 可能标注"效率"，但 voiceover 中是"工作效率"
- **应对方案**：模糊匹配，允许部分匹配
- **降级方案**：如果找不到，跳过这个词

**风险 3：时间轴计算误差累积**
- **原因**：TTS timestamps 可能有 ±0.1 秒误差
- **应对方案**：允许 ±0.3 秒误差，用户感知不明显
- **降级方案**：如果误差 > 0.5 秒，在日志中标记，人工审核

## 8. 最终决策

### 8.1 采用方案

**采用技术推荐的"最小化方案"，分 3 个阶段实施。**

**第一阶段（MVP，5 天）**：
- 只做 `emphasis_words` 标注
- 时间轴由 TTS timestamps 计算
- 验收标准：成功率 ≥ 90%，修改率下降 ≥ 5%

**第二阶段（节奏规则，3 天）**：
- 根据 Scene 类型应用预设节奏规则
- 验收标准：修改率下降 ≥ 10%

**第三阶段（编辑器，5 天）**：
- 前端提供时间轴编辑器
- 验收标准：使用率 ≥ 20%

### 8.2 不采用的方案

**不采用"规则方案"（技术方案 B）**：
- 原因：规则方案效果上限低，无法适应内容差异
- 妥协：如果 MVP 失败，再考虑规则方案

**不采用"后处理方案"（技术方案 C）**：
- 原因：需要用户手动操作，违背"自动生成"的产品定位
- 妥协：作为第三阶段的补充功能

### 8.3 失败退出机制

**第一阶段失败**：
- 如果 LLM 成功率 < 90%，回退到规则方案
- 如果用户感知 < 60%，调整动效参数，重新测试
- 如果修改率没下降，停止开发，承认假设错误

**第二阶段失败**：
- 如果用户感知 < 50%，说明规则不够灵活，需要 LLM 生成节奏数据
- 如果修改率没下降，停止开发

**第三阶段失败**：
- 如果使用率 < 20%，说明用户不需要编辑器，停止开发

## 9. 产品价值坚持

### 9.1 为什么必须做

**产品层面**：
- 当前系统的核心问题是"内容对但不抓人"
- 不解决"表达"和"节奏"问题，产品永远是"效率工具"
- 这是产品从"能用"到"好用"的关键

**业务层面**：
- 用户修改率 30% 是真实数据（来自日志）
- 即使"60% 是因为节奏不对"是推测，但修改率高是事实
- 降低修改率是提升用户满意度的最直接方法

**竞争层面**：
- 普通 AI 视频工具都在做"内容生成"
- "重点词强调"和"节奏起伏"是差异化的关键
- 如果不做，产品没有护城河

### 9.2 如果不做会怎样

- 视频永远是"机器念稿"，无法像人说话
- 关键信息无法被强调，观众记不住
- 节奏永远是匀速播放，无法制造起伏
- 用户会觉得"内容对，但不抓人"
- 产品会停留在"能用但不好用"

### 9.3 对技术的最后陈述

**我接受技术的批评**：
- v1 方案确实过度设计
- 让 LLM 生成时间轴确实是错的
- 验收标准确实不够可测

**但我不接受"不该做"的结论**：
- "重点词强调"不是过度设计，是产品差异化的关键
- 技术的"最小化方案"很好，我已经采纳
- 但如果技术认为"任何新功能都有风险，所以不该做"，那产品无法迭代
